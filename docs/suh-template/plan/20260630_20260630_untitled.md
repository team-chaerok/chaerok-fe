# 🎯 [Network] 백엔드 API 공통 요청 함수 및 Request/Response 구조 전략 문서

> Issue: [#8 백엔드 API 공통 요청 함수 및 Request/Response 구조 구현](https://github.com/team-chaerok/chaerok-fe/issues/8)

---

## 1. 요약

Dio 기반 HTTP 클라이언트를 구성하고, 인증 토큰 자동 주입 / 공통 에러 처리 인터셉터와 `ApiResponse<T>` 래퍼 모델을 도입하여, 모든 서비스 레이어에서 일관된 방식으로 백엔드 API를 호출할 수 있는 네트워크 공통 계층을 구축한다.

---

## 2. 배경 및 목적

**문제/필요성**
- API 호출마다 baseUrl, 헤더, 토큰 주입 코드가 중복될 가능성
- Response/Error 구조가 정의되지 않아 각 서비스마다 파싱 로직이 달라질 위험

**목표**
- 모든 HTTP 호출이 `DioClient`를 통해 통일된 방식으로 처리
- 인증 토큰을 자동으로 헤더에 주입하고, 만료 시 자동 재발급
- 성공/실패 응답을 `ApiResponse<T>` / `ApiError`로 일관되게 반환

**범위**
- 포함: Dio 클라이언트, Auth 인터셉터, Error 인터셉터, ApiResponse, ApiError 모델
- 제외: 각 도메인 Repository/Service 구현, 구체적인 API 엔드포인트 정의

---

## 3. 요구사항

**필수 (P0)**
- `DioClient`: baseUrl, timeout, 인터셉터 설정 + `get/post/put/delete` 공통 메서드
- `AuthInterceptor`: flutter_secure_storage에서 access token 읽어 `Authorization: Bearer` 헤더 주입
- `ErrorInterceptor`: 네트워크 오류 / 타임아웃을 `ApiError`로 변환
- `ApiResponse<T>`: 공통 응답 래퍼 모델
- `ApiError`: 에러 정보 모델

**중요 (P1)**
- 401 응답 시 refresh token으로 자동 재발급 후 원래 요청 재시도
- 재발급 실패 시 로그아웃 이벤트 발행 (콜백 또는 스트림)
- `AppSecrets`에 `baseUrl` 상수 추가

**선택 (P2)**
- `ApiResponse` 직렬화에 `json_serializable` / `freezed` 도입 (현재 이슈 범위 밖)

---

## 4. 선택한 접근 방식

**방식**: 수동 직렬화 + Dio Interceptor 체인

**이유**
- 현재 `pubspec.yaml`에 `build_runner`, `freezed`, `json_serializable` 의존성이 없어 추가 설정 비용 없이 빠르게 구현 가능
- `ApiResponse<T>`의 구조가 단순하므로 수동 `fromJson`으로 충분
- 인터셉터 체인(Auth → Error)이 명확히 분리되어 테스트/교체 용이

**대안 고려**: `freezed + json_serializable`
- 보일러플레이트 감소, 불변 모델 장점 있음
- 단, build_runner 설정 필요 — 향후 모델이 많아지면 전환 권장

---

## 5. 주요 결정사항

| 결정 | 선택 | 이유 |
|------|------|------|
| HTTP 클라이언트 | Dio | 이슈 명시, 인터셉터 지원 우수 |
| 토큰 저장 | flutter_secure_storage | 암호화 저장, iOS Keychain / Android Keystore |
| baseUrl 관리 | AppSecrets에 상수 추가 | 기존 소셜 키와 같은 위치로 일관성 확보 |
| 모델 직렬화 | 수동 fromJson | 패키지 추가 없이 빠른 구현, 이후 freezed 전환 용이 |
| 싱글톤 구성 | DioClient 클래스 (static instance) | 앱 전체에서 동일 인스턴스 사용 |

## 6. 파일 구조 및 구현 명세

```
lib/
├── core/
│   ├── config/
│   │   └── app_secrets.dart          # baseUrl 상수 추가
│   └── network/
│       ├── dio_client.dart            # DioClient (싱글톤)
│       └── interceptors/
│           ├── auth_interceptor.dart  # 토큰 주입 + 401 재발급
│           └── error_interceptor.dart # 공통 에러 변환
└── data/
    └── models/
        ├── api_response.dart          # ApiResponse<T>
        └── api_error.dart             # ApiError
```

### pubspec.yaml 추가 의존성

```yaml
dependencies:
  dio: ^5.8.0
  flutter_secure_storage: ^9.2.4
```

### AppSecrets 추가

```dart
static const String baseUrl = 'https://api.chaerok.com'; // TODO: 실제 URL 확인 필요
```

### DioClient 인터페이스

```dart
class DioClient {
  static final DioClient instance = DioClient._internal();
  
  Future<ApiResponse<T>> get<T>(String path, {Map<String, dynamic>? queryParameters, T Function(dynamic)? fromJson});
  Future<ApiResponse<T>> post<T>(String path, {dynamic data, T Function(dynamic)? fromJson});
  Future<ApiResponse<T>> put<T>(String path, {dynamic data, T Function(dynamic)? fromJson});
  Future<ApiResponse<T>> delete<T>(String path, {T Function(dynamic)? fromJson});
}
```

### ApiResponse<T> 구조

```dart
class ApiResponse<T> {
  final bool success;
  final T? data;
  final String? message;
  final int statusCode;
}
```

### ApiError 구조

```dart
class ApiError {
  final int statusCode;
  final String message;
  final String? errorCode;
}
```

### AuthInterceptor 로직

```
onRequest:
  1. flutter_secure_storage에서 access_token 읽기
  2. 존재 시 'Authorization: Bearer {token}' 헤더 추가

onError (401):
  1. flutter_secure_storage에서 refresh_token 읽기
  2. POST /auth/refresh 요청
  3. 성공: 새 access_token 저장 → 원래 요청 재시도 (requestOptions.extra로 재진입 방지)
  4. 실패: 토큰 전체 삭제 → 로그아웃 콜백 호출
```

## 7. 고려사항 및 위험요소

**기술적 위험**
- **백엔드 API Response 스펙 미확정**: `ApiResponse<T>`의 JSON 키 구조(`success`, `data`, `message`)는 백엔드 스펙 확정 후 조정 필요
  - 대응: 스펙 미확정 상태에서는 fromJson을 유연하게 null-safe 처리
- **401 재발급 중 동시 요청**: 여러 요청이 동시에 401 받으면 refresh가 중복 호출될 수 있음
  - 대응: `Completer`로 재발급 중 다른 요청은 대기 처리
- **flutter_secure_storage iOS 시뮬레이터 이슈**: 시뮬레이터에서 keychain 접근 불안정할 수 있음
  - 대응: `iOptions: IOSOptions(accessibility: IOSAccessibility.first_unlock)` 설정

**비즈니스 위험**
- 로그아웃 이벤트 처리: `DioClient`에서 직접 Navigator 접근 불가 → 콜백 또는 스트림으로 상위 전달 필요
