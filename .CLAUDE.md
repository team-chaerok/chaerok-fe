# CLAUDE.md

이 문서는 Chaerok Flutter 프로젝트에서 Claude가 코드를 작성하거나 수정할 때 반드시 따라야 하는 작업 규칙이다.

Claude는 작업 전 이 문서를 기준으로 프로젝트 구조, 네이밍, 디자인 시스템, lint 규칙, 코드 작성 방식을 판단한다.

---

## 1. 프로젝트 개요

Chaerok은 Flutter 기반 앱 프로젝트이다.

핵심 컨셉은 위치 기반 관광/코스 추천, 촬영, 앨범, 엽서 생성 흐름을 가진 모바일 앱이다.

주요 기능 단위는 다음과 같다.

```text
home        → 현재 위치 기반 추천, 메인 화면
course      → 채록길 목록, 상세, 코스 장소
place       → 관광지/장소 목록, 상세
camera      → 필름 카메라 촬영
album       → 촬영 사진, 현상 결과
postcard    → 여행 종료 후 엽서 생성 결과
mypage      → 사용자 설정 및 마이페이지
onboarding  → 서비스 소개 및 권한 안내
splash      → 앱 진입 및 초기 분기
```

---

## 2. 기본 작업 원칙

Claude는 다음 원칙을 우선한다.

1. 기존 프로젝트 구조를 유지한다.
2. 불필요한 폴더나 추상화를 만들지 않는다.
3. 화면 파일에 API 호출이나 복잡한 비즈니스 로직을 직접 작성하지 않는다.
4. 재사용 가능한 UI는 `shared/widgets/`에 둔다.
5. 특정 기능 전용 UI는 해당 `features/{feature}/widgets/`에 둔다.
6. 색상, 폰트, 간격, radius는 디자인 토큰을 우선 사용한다.
7. 새 코드는 `flutter analyze`를 통과해야 한다.
8. 기존 파일을 수정할 때는 주변 코드 스타일을 따른다.
9. 사용하지 않는 import, 변수, 함수는 남기지 않는다.
10. 임의로 패키지를 추가하지 않는다. 패키지 추가가 필요한 경우 이유를 명확히 설명한다.

## 3. 프로젝트 구조

기본 구조는 다음과 같다.

```text
lib/
├── main.dart
├── app/
├── core/
├── shared/
├── features/
└── data/
```

각 폴더의 역할은 다음과 같다.

```text
app      → 앱 실행, 라우팅, 전역 초기 설정
core     → 앱 전체 기반 코드
shared   → 여러 기능에서 재사용되는 공통 요소
features → 화면과 기능 단위 코드
data     → API, 로컬 저장소, DTO 등 데이터 연결부
```

자세한 구조 규칙은 다음 문서를 따른다.

```text
docs/project-structure.md
```

## 4. 폴더 배치 규칙

파일 위치는 다음 기준으로 판단한다.

```text
특정 기능에만 사용된다 → features/{feature}/
여러 기능에서 재사용된다 → shared/
앱 전체 기반 코드다 → core/
API, 로컬 저장소, 응답 DTO다 → data/
라우팅 또는 앱 초기 설정이다 → app/
```

예시:

```text
CourseDetailScreen       → features/course/screens/
CourseCard               → features/course/widgets/
ChaerokButton            → shared/widgets/
LocationService          → shared/services/
ApiClient                → core/network/
ChaerokTheme             → core/design_system/
TourPlaceResponse        → data/dto/
TourApiDatasource        → data/datasources/
```

## 5. Feature 내부 구조

각 feature는 필요에 따라 다음 구조를 사용한다.

```text
features/{feature_name}/
├── screens/
├── widgets/
├── models/
├── controllers/
└── repositories/
```

단, 필요 없는 폴더는 만들지 않는다.

예를 들어 단순 화면만 있는 기능이면 다음 구조도 허용한다.

```text
features/splash/
└── screens/
    └── splash_screen.dart
```

## 6. 화면과 로직 분리 규칙

화면은 UI 표현에 집중한다.

권장 흐름은 다음과 같다.

```text
Screen
→ Controller
→ Repository
→ Datasource
→ API / Local Storage
```

예시:

```text
CourseDetailScreen
→ CourseController
→ CourseRepository
→ TourApiDatasource
```

화면 파일에서 직접 하지 말아야 할 것:

```text
API 직접 호출
로컬 저장소 직접 접근
복잡한 데이터 변환
긴 조건 분기
외부 SDK 제어 로직 직접 작성
```

---

## 7. DTO와 Model 분리 규칙

API 응답 DTO와 화면에서 사용하는 모델은 분리한다.

```text
data/dto/course_response.dart
features/course/models/course.dart
```

규칙:

1. DTO는 외부 API 응답 구조에 맞춘다.
2. 화면에서는 DTO를 직접 사용하지 않는다.
3. Repository에서 DTO를 feature model로 변환한다.

예시:

```dart
Course toCourse(CourseResponse response) {
  return Course(
    id: response.contentId,
    name: response.title,
    address: response.addr1,
  );
}
```

## 9. 디자인 토큰 사용 규칙

화면에서 색상, 폰트 크기, 여백, radius를 직접 반복 작성하지 않는다.

지양:

```dart
Container(
  padding: const EdgeInsets.all(16),
  decoration: BoxDecoration(
    color: Color(0xFFFFFFFF),
    borderRadius: BorderRadius.circular(12),
  ),
)
```

권장:

```dart
Container(
  padding: const EdgeInsets.all(ChaerokSpacing.md),
  decoration: BoxDecoration(
    color: ChaerokColors.surface,
    borderRadius: BorderRadius.circular(ChaerokRadius.md),
  ),
)
```

사용 기준:

```text
색상      → ChaerokColors
폰트      → ChaerokTypography 또는 Theme.of(context).textTheme
간격      → ChaerokSpacing
Radius    → ChaerokRadius
Shadow    → ChaerokShadows
ThemeData → ChaerokTheme
```

## 10. 공통 위젯 규칙

공통 위젯은 `shared/widgets/`에 둔다.

예시:

```text
shared/widgets/chaerok_button.dart
shared/widgets/chaerok_app_bar.dart
shared/widgets/chaerok_card.dart
shared/widgets/loading_view.dart
shared/widgets/empty_view.dart
```

공통 위젯 판단 기준:

```text
여러 화면에서 반복 사용된다.
앱 전역 UI 패턴이다.
디자인 수정 시 여러 화면에 영향을 준다.
특정 feature에 종속되지 않는다.
```

특정 feature 전용 위젯은 해당 feature 내부에 둔다.

```text
features/course/widgets/course_card.dart
features/camera/widgets/film_frame_overlay.dart
features/postcard/widgets/postcard_preview.dart
```

## 11. 네이밍 규칙

파일명은 `snake_case`를 사용한다.

```text
course_detail_screen.dart
course_card.dart
chaerok_button.dart
chaerok_colors.dart
```

클래스명은 `PascalCase`를 사용한다.

```dart
class CourseDetailScreen extends StatelessWidget {}
class CourseCard extends StatelessWidget {}
class ChaerokButton extends StatelessWidget {}
```

변수명과 함수명은 `camelCase`를 사용한다.

```dart
final courseList = <Course>[];

void fetchCourseList() {}
```

디자인 시스템 클래스와 공통 위젯은 `Chaerok` prefix를 사용한다.

```text
ChaerokColors
ChaerokTypography
ChaerokSpacing
ChaerokRadius
ChaerokShadows
ChaerokTheme
ChaerokButton
ChaerokAppBar
ChaerokCard
```

## 12. Import 규칙

package import를 우선 사용한다.

권장:

```dart
import 'package:chaerok/features/course/models/course.dart';
```

지양:

```dart
import '../../models/course.dart';
```

`directives_ordering` lint를 따른다.

즉, import는 알파벳순으로 정렬한다.

예시:

```dart
import 'package:chaerok/core/design_system/chaerok_colors.dart';
import 'package:chaerok/core/design_system/chaerok_radius.dart';
import 'package:chaerok/core/design_system/chaerok_spacing.dart';
import 'package:flutter/material.dart';
```

## 13. Lint 규칙

프로젝트는 `analysis_options.yaml` 규칙을 따른다.

중요 규칙:

```text
avoid_print
prefer_const_constructors
prefer_final_locals
use_build_context_synchronously
avoid_relative_lib_imports
directives_ordering
require_trailing_commas
sort_child_properties_last
```

Claude는 코드를 작성한 뒤 다음 기준을 만족해야 한다.

```text
사용하지 않는 import 제거
가능한 const 추가
async gap 이후 BuildContext 안전성 확인
relative import 사용 지양
trailing comma 유지
Flutter widget child 속성은 마지막에 배치
```

---

## 14. 코드 작성 스타일

위젯 작성 시 다음 기준을 따른다.

1. `const` 생성자를 가능한 사용한다.
2. `build` 메서드가 길어지면 private method 또는 별도 widget으로 분리한다.
3. 같은 UI가 2번 이상 반복되면 widget으로 분리한다.
4. 상태가 없는 UI는 `StatelessWidget`을 우선 사용한다.
5. 임시 디버깅 로그로 `print`를 사용하지 않는다.
6. magic number는 디자인 토큰 또는 상수로 분리한다.
7. null safety를 유지한다.
8. nullable 값은 명시적으로 처리한다.

## 15. BuildContext 사용 규칙

`async` 이후 `BuildContext`를 사용할 때는 mounted 여부를 확인한다.

지양:

```dart
onPressed: () async {
  await controller.fetchData();
  Navigator.of(context).pop();
}
```

권장:

```dart
onPressed: () async {
  await controller.fetchData();

  if (!context.mounted) return;

  Navigator.of(context).pop();
}
```

## 16. 상태관리 규칙

현재 상태관리 방식이 명확히 고정되지 않은 경우 `controllers/`를 사용한다.

```text
features/course/controllers/course_controller.dart
```

Controller의 역할:

```text
화면 상태 관리
로딩/에러/성공 상태 처리
사용자 이벤트 처리
Repository 호출
```

Controller에서 하지 말아야 할 것:

```text
직접 HTTP 요청 작성
DTO를 화면에 그대로 노출
복잡한 UI 위젯 생성
```

## 17. API 연결 규칙

API 연결은 다음 흐름을 따른다.

```text
Feature Repository
→ DataSource
→ ApiClient
```

예시:

```text
CourseRepository
→ TourApiDatasource
→ ApiClient
```

규칙:

1. Screen에서 datasource를 직접 호출하지 않는다.
2. Screen에서 API URL을 직접 다루지 않는다.
3. Repository에서 DTO를 화면 모델로 변환한다.
4. API 에러는 공통 에러 타입 또는 결과 타입으로 정리한다.

## 20. 작업 전 확인 명령어


## 21. 금지 사항

Claude는 다음 작업을 임의로 수행하지 않는다.

```text
프로젝트 구조를 임의로 대규모 변경하지 않는다.
사용자 요청 없이 패키지를 추가하지 않는다.
analysis_options.yaml 규칙을 임의로 완화하지 않는다.
lint 오류를 ignore로 우회하지 않는다.
화면에서 API를 직접 호출하지 않는다.
디자인 토큰 없이 색상/폰트/여백 값을 반복 작성하지 않는다.
기능 전용 위젯을 shared에 무분별하게 넣지 않는다.
공통 위젯을 feature마다 중복 작성하지 않는다.
DTO를 화면에 직접 노출하지 않는다.
불필요한 abstraction, interface, usecase를 만들지 않는다.
```

## 22. Claude 응답 규칙

Claude가 코드 수정 작업을 수행할 때는 다음 형식으로 정리한다.

```text
1. 수정한 파일
2. 변경 내용 요약
3. 구조 규칙 준수 여부
4. 실행해야 할 검증 명령어
```

예시:

```text
수정한 파일:
- lib/features/course/screens/course_list_screen.dart
- lib/features/course/widgets/course_card.dart

변경 내용:
- 코스 목록 화면 추가
- 코스 카드 위젯 분리
- ChaerokSpacing, ChaerokColors 기반 스타일 적용

검증:
- dart format lib test
- flutter analyze
```

## 23. 최종 기준

Claude는 모든 작업에서 다음 기준을 최우선으로 따른다.

```text
기능 전용이면 features
재사용 요소면 shared
앱 기반이면 core
데이터 연결이면 data
앱 설정이면 app
```

디자인 관련 작업은 다음 기준을 따른다.

```text
디자인 토큰은 core/design_system
공통 UI는 shared/widgets
기능 전용 UI는 features/{feature}/widgets
앱 전체 테마는 ChaerokTheme
```

코드 품질 기준은 다음 명령어 통과를 목표로 한다.

```bash
dart format lib test
flutter analyze
flutter test
```
